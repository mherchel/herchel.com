Ternary Operators in twig

Only adds the has-content-right if the page.content_right region has content.

You can also add the else statement, which evaluates if the condition is false:

<div class="layout-container {{ page.content_right ? 'has-content-right': 'no-content-right'}}">

You can add an else statement in the for loop that will resolve if there are no items.

<ul>
  {% for item in items %}
    <li>{{ item.content }}</li>
  {% else %}
    <li>This list is empty</li>
  {% endfor %}
</ul>

You also have access to a loop variable within the loop.

  • loop.index The current iteration of the loop. (1 indexed)
  • loop.index0 The current iteration of the loop. (0 indexed)
  • loop.revindex The number of iterations from the end of the loop (1 indexed)
  • loop.revindex0 The number of iterations from the end of the loop (0 indexed)
  • loop.first True if first iteration
  • loop.last True if last iteration
  • loop.length The number of items in the sequence
  • loop.parent The parent context
<ul class='blog-post__tags field__items'>

Debugger Breakpoints

Native to Twig:

or

_context references the current context and contains all variables passed to the template such as variables sent from theme(), prepared by preprocess, or set in the template. Adding {{ dump() }} without specifying a variable is equivalent to {{ dump(_context) }}.

with Devel module:

Add classes in Drupal twig

  • Printing <div{{ attributes }}> works fine.
  • Don‚Äôt add additional spacing in the attributes.

  • Remove class

]

]
%}

Render API

  1. Structured arrays with data and hints on how to render it.
  2. Pipeline: process Drupal goes through to serve a request.

Render Arrays

  • Properties are array items where the key‚Äôs names start with a hash. They determine how an instance of an element will ultimately be rendered.

  • Elements are items that do not start with a hash. They can have children. It‚Äôs an individual section of the array and represent data.

  • #markup Provides HTML directly

  • #theme Provides information on which template is needed to generate the HTML (and where to find it).

  • #type Shorthand for writing a more complex element.

Value properties

Store strings, integers, booleans, arrays. Examples:

  • #prefix
  • #suffix
  • #title
  • #weight

Callable properties

Less common. These hold references to PHP callables. At specific points in the rendering, these can be called to alter the state.

  • #pre_render
  • #post_render

Composer

  • composer install - initial install. If a composer.lock file exists, it will ignore the package.json and use the lock file.

  • composer update - Updates with the latest tags off of the package.json file.

    • You can use composer update [package] to only update a single package.
  • composer require - installs new package (or module), and writes it to the package.json.

Multiple ways to check if field exists in twig

Sometimes doing a simple {% if content.xxx %} doesn‚Äôt correctly check. There‚Äôs an 8 year old (!!!) issue on Drupal.org (link) that details the issues. Mark Carver‚Äôs comment provides some insight on why this is not yet fixed.

In the meantime, there are multiple ways to do conditionals, that will work around this bug.

This is the normal way to do things.

{% if content.field %}
  <div class="field">
    {{ content.field }}
  </div>
{% endif %}

If that doesn‚Äôt work you can render it

{% if content.field|render %}
  <div class="field">
    {{ content.field }}
  </div>
{% endif %}

You can also check the node object, which has the getValue() method, which can be reference in twig by {{ node.field.value }}. This has the added benefit of not using the field template (and associated markup).

{% if node.field.value %}
  <div class="field">
    {{ node.field.value }}
  </div>
{% endif %}

If needed you can strip tags and trim it after render

{% if content.field|render|striptags|trim %}
  <div class="field">
    {{ content.field }}
  </div>
{% endif %}

You can also use twig syntax similar to the following if needed

{% if content.field|render|striptags|trim is not empty %}

Getting fields (and entity reference fields) within preprocess

Thanks to Marcos Cano for helping me out with all of this!

When debugging in PHPStorm, it‚Äôs really useful to use the Evaluate panel. Prior to this, I was using PHPStorm‚Äôs console tab, but that didn‚Äôt always work!

Once stopped at a breakpoint, open up the Evaluate panel by right-clicking on the item you want to start with, and select ‚ÄúEvaluate Expression‚Äù.

You can get access to all of the fields by using Entity API Methods.

First load up the node object (note I didn‚Äôt yet test this code)

$variables['node']

Then do a getFields() on it to see the fields

$variables['node']->getFields();

You‚Äôll see an array of stuff including fields.

From there you can use the get() method to get the field

$variables['node']->get('field_firstname');

but you can also use what Marcos Cano called magic by using the fieldname as the method. It makes things more readable.

$variables['node']->field_firstname;

From there you can try a couple different things based on the field type. Sometimes you can get the value by doing

$variables['node']->field_firstname->getValue();

A lot of times, you can browse down the object, and look for the key that holds the actual data that you‚Äôre looking for.

This might be uri if it‚Äôs a link field. Or, it might be value if it‚Äôs a text field. Expand the array in the watch expression and take a look.

// a link field might look like
$variables['node']->field_firstname->uri;
// or if its a text field
$variables['node']->field_firstname->value;

Loading field from an entity reference.

You‚Äôll probably need to load some data from an entity reference. Here‚Äôs how to do it.

Use the above methods to bring up the field for the reference:

$variables['node']->field_reference

Then you need to open the entity object(?). This won‚Äôt show when you‚Äôre looking in the watches panel for whatever reason.

$variables['node']->field_reference->entity

From there you can use the same methods to access the data as we did before.

$variables['node']->field_reference->entity->field_firstname->value

And you can even go into different entity references. Here‚Äôs an example that I worked with. It‚Äôs a reference to another node, that has a reference to a file!

$variables['node']->field_parent->entity->field_show_media->entity->field_media_image->entity->uri->value

Cachability and Twig

Drupal 8 has a new caching system that really sets above from other systems. However there‚Äôs a ~bug~ idiosyncrasy in the way twig templates handle caching.

If you‚Äôre not printing out your templates with the {{ content }} variable. Say

<div{{ attributes }}>
  {{ content.first_name }} {{ content.last_name }}
</div>

the cacheable metadata won‚Äôt be included. This means that if your data changes within the database, the theming system won‚Äôt know to show the newer data.

The temporary fix is straightforward, add the following line to the bottom of you twig file:

{% set catch_cache = content|render %}

Thanks to Avi Schwab for finding this solution and Mark Conroy for putting it somewhere I actually noticed it.

Check for permissions in Twig

You can check for permissions in twig by using something like

{% if user.hasPermission('create prospect content') %}
  <a class="button--add-team-member" href="/node/add/prospect">Add New Prospect</a>
{% endif %}

But, how do we know what the value for the hasPermission() function should be?

The easiest way is to navigate to the admin permissions page, and inspect the checkbox for the permission you‚Äôd like to check for. Within there, you‚Äôll see a name attribute that has the information you‚Äôre looking for.

Get properly formatted URL from link field within twig

If your link field has an internal path (eg /path/foo), if you output node.field_link.uri in your twig template, it‚Äôll output internal:/path/foo into your markup. We need to convert this into a proper URL.

Within preprocess, you can get this with

$variables['node']->field_link->first()->getUrl()->toString()

Within twig, you can simply output

{{ node.field_link.0.url }}

According to this StackOverflow post we do not need to include toString within twig because ‚Äútwig will cast this object as a string this will call the magic method toString() and will output the url as a string value.‚Äù
